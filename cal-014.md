# Summary

The codebase is in **good structural shape** post CAL-015.

## Strengths

- Clear architecture: services → stores → features → components
- Strong composable extraction from CAL-015
- Utility layer is mostly well-defined
- Component responsibilities improved overall
- No evidence that the app needs architectural rewrites

## Core Issues Found

The main issues identified during CAL-014 were:

- responsibility hotspots
- duplicated UI logic
- source-of-truth drift
- a few legacy artifacts
- repeated staff identity / avatar presentation logic

## Main Conclusion

> The codebase needs **consolidation and consistency**, not restructuring.

# Key Findings

## 1. Legacy `StaffList.vue`

**Finding**

- `StaffList.vue` appeared to be a legacy artifact
- It was not part of the active management flow
- It duplicated avatar loading logic
- It used older patterns and weak keys

**Result**

- Verified unused
- Removed later in CAL-027

**Status**

- Resolved

---

## 2. Event type source-of-truth drift

**Files involved**

- `eventTypeConfig.ts`
- `selectOptions.ts`
- `CalendarHeaderFilters.vue`

**Finding**

- Event types were defined in multiple places
- Filterable types were hardcoded in component code
- This created drift risk across form/config/filter UI

**Result**

- Unified later in CAL-028
- Shared `eventTypes` export added
- Duplicate event type list removed from `selectOptions.ts`
- Header filters updated to use shared event types

**Status**

- Resolved

---

## 3. Repeated event date formatting

**Files involved**

- `formatEventDates.ts`
- `EventModal.vue`
- `DayModal.vue`

**Finding**

- Shared date-formatting utility already existed
- Active modals still duplicated local formatting logic

**Result**

- Centralized later in CAL-030
- `EventModal.vue` and `DayModal.vue` now use shared date formatting helper

**Status**

- Resolved

---

## 4. Repeated modal shell patterns

**Pairs**

- `NewEventModal.vue` ↔ `EditEventModal.vue`
- `DayModal.vue` ↔ `EventModal.vue`

**Finding**

- Repeated shell/header structure and styling
- Repeated title display logic in event-detail modals

**Result**

- Addressed partially in CAL-029
- Event form modal shell naming aligned
- Event title display moved into computed values in both event detail modals

**Status**

- Partially resolved
- Further consolidation may still be possible, but no broader abstraction was introduced

---

## 5. Repeated staff identity / avatar presentation

**Seen in**

- `CalendarEvent.vue`
- `EventModal.vue`
- `DayModal.vue`
- `CalendarHeaderFilters.vue`

**Finding**

- Avatar loading, color styling, visible-count patterns, and summary patterns were repeated
- Identity presentation is important in this app, so inconsistency risk is meaningful

**Status**

- Still open
- Remains a valid follow-up concern after CAL-014

## 6. `CalendarBody.vue` as a major orchestration hotspot

**Finding**

- `CalendarBody.vue` still owns substantial coordination logic:
  - calendar grid construction
  - month cell generation
  - event assignment caching
  - week/day mapping
  - spanning-event modal orchestration
  - viewport resize handling

**Assessment**

- Not a bug
- Not an architectural failure
- But still one of the clearest refinement hotspots

**Status**

- Open
- Worth monitoring, but not urgent

# Final Assessment

## What CAL-014 established

- The project does **not** need a major refactor
- The main cleanup work should be:
  - targeted
  - incremental
  - scoped by concrete duplication or drift
- The codebase is best improved through small PRs, not sweeping rewrites

## Current project state after follow-up work

- legacy cleanup has started successfully
- event type drift has been reduced
- modal consistency has improved
- shared date formatting is now in use
- the remaining work is narrower and lower-risk than it was at the start of CAL-014

# Recommendations Going Forward

## Good next candidates after CAL-014

- standardize staff identity presentation
